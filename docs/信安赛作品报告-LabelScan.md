# LabelScan：面向 Label-Only 黑盒模型的成员隐私风险审计系统作品报告

## 摘要

随着机器学习服务在图像识别、内容审核、医疗辅助诊断、金融风控等场景中广泛应用，训练数据隐私泄露逐渐成为人工智能安全领域的重要问题。成员推理攻击关注的是：给定一个样本和目标模型的查询接口，攻击者能否判断该样本是否曾参与目标模型训练。即使模型只返回最终分类标签，不暴露概率分数、梯度、参数或中间特征，已有研究仍表明模型训练过程可能留下可被利用的成员化痕迹。

LabelScan 是一款面向 Label-Only 黑盒场景的模型成员隐私风险审计系统。系统以“只可观察分类标签”为核心约束，将影子模型行为信号与决策边界几何信号结合起来，对待测样本进行成员风险评估。影子模型路径通过交叉熵损失刻画样本是否呈现类似训练成员的行为特征；边界攻击路径通过 HopSkipJump 思想估计样本到目标模型决策边界的距离，并结合局部扰动稳定性分析样本是否处于模型更熟悉的决策区域。系统进一步设计动态阈值标定和双轨融合判定机制，避免固定阈值在不同模型、不同数据分布下产生较大偏差。

在工程实现上，LabelScan 采用 Go 与 Python 解耦架构。Python 模型服务负责加载目标模型和影子模型，并提供标准化 HTTP 推理接口；Go 审计后端负责样本调度、边界搜索、并发查询、风险融合、任务队列、取消控制和报告生成；Web 控制台负责参数配置、服务健康检查、任务提交、状态轮询和结果展示。系统支持服务器端常驻运行，模型服务仅在本机或容器内网暴露，用户端可通过 Windows 启动器访问受保护的 HTTPS 控制台，无需安装 Docker、Python 或模型文件。

实验与测试表明，LabelScan 能在严格 Label-Only 条件下输出样本级风险等级、证据指标和可复核报告。在内置 CIFAR-10 示例模型上，系统基于 10000 个样本完成影子模型阈值统计，成员侧识别准确率约为 64.8%；在标准抽样审计中，系统能够给出查全率约 84%、查准率约 63% 的成员风险识别结果。该作品将成员推理攻击从理论验证转化为可部署、可交互、可复核的隐私审计工具，可用于模型上线前自查、外部模型服务隐私风险评估和人工智能安全教学演示。

关键词：成员推理攻击；Label-Only；黑盒模型；隐私审计；边界攻击；影子模型；人工智能安全

## 第一章 作品概述

### 1.1 研究背景

机器学习模型在训练过程中会对训练数据形成不同程度的记忆。对于图像分类模型而言，训练集样本通常会处于模型更稳定、更高置信或更远离决策边界的区域。成员推理攻击利用这种差异，尝试判断某个样本是否属于目标模型训练集。该问题在隐私敏感场景中具有现实意义：若攻击者能够确认某条医疗影像、身份照片或行为记录曾被用于训练模型，就可能间接暴露用户参与过某项业务、疾病筛查或数据采集活动。

传统成员推理方法多依赖模型输出概率、置信度或 logits。现实中的模型服务为了降低泄露风险，往往只返回最终分类标签。例如，图像审核接口可能只返回“合格/违规”，识别服务可能只返回类别名称。这类场景被称为 Label-Only 黑盒场景。由于攻击者无法获得概率分数，许多基于置信度阈值或攻击分类器的方法无法直接使用，但模型决策边界仍可能保留成员样本与非成员样本之间的几何差异。因此，Label-Only 条件下的成员隐私审计更贴近真实模型服务的安全边界。

LabelScan 面向这一问题构建完整审计系统。作品不是简单展示攻击结果，而是围绕“如何让模型拥有者或安全审计人员在授权条件下复核模型是否泄露训练成员信息”这一目标，形成从模型接入、样本抽取、阈值标定、风险计算、任务管理到报告生成的完整闭环。

### 1.2 问题定义

设目标分类模型为 \(f\)，输入样本为 \(x\)，真实类别或参考类别为 \(y\)。审计人员只能通过查询接口获得模型返回的分类标签 \(f(x)\)，无法访问训练集、模型参数、梯度、概率分数或中间层特征。成员推理审计要解决的问题是：

\[
M(x, f) \in \{\text{member risk}, \text{non-member risk}\}
\]

即判断样本 \(x\) 是否呈现出更接近训练成员的风险特征。LabelScan 不把输出解释为法律意义或事实意义上的“确定属于训练集”，而是输出成员风险等级及证据指标，包括影子模型 loss、平均边界距离、扰动波动系数和融合判定结论。该设计更适合安全审计场景，因为审计结果需要可解释、可复核，并能体现证据来源。

### 1.3 作品目标

LabelScan 的设计目标包括以下几个方面。

第一，支持严格 Label-Only 黑盒审计。系统默认只依赖目标模型返回的分类标签，能够在目标模型不暴露概率分数、梯度和参数的条件下进行边界风险分析。

第二，支持样本级成员风险评估。系统不仅输出整体统计指标，还为每个样本生成风险等级和证据摘要，使审计人员能够查看具体样本为何被判为高风险或低风险。

第三，提供双轨证据融合能力。系统同时利用影子模型行为信号和目标模型边界几何信号，降低单一指标受到样本难度、模型精度、查询预算或数据分布偏移影响时产生的误判。

第四，具备工程化交付能力。系统包含 Web 控制台、RESTful 接口、报告生成、任务队列、取消机制、后台自动拉起脚本和 Windows 启动器，能够在服务器端部署后供普通用户通过浏览器或 exe 入口使用。

第五，考虑安全部署边界。目标模型服务与影子模型服务只在受控网络中开放，由 Go 后端统一调度；公网仅暴露 Web 控制台或反向代理入口，并可叠加访问控制和限流策略，减少模型接口被直接刷取的风险。

### 1.5 应用场景

LabelScan 可应用于以下场景。

一是模型上线前隐私自查。模型提供者在部署分类模型前，可抽取训练集样本和未训练样本进行审计，判断模型是否存在明显成员泄露风险。

二是第三方安全评估。在授权前提下，审计人员可接入目标模型 API，对目标服务进行 Label-Only 条件下的黑盒隐私风险评估。

三是数据治理与合规辅助。对于涉及敏感数据的模型，LabelScan 可以作为隐私风险评估工具之一，为数据使用、模型训练和服务上线提供技术依据。

四是人工智能安全教学与竞赛展示。系统从攻击原理到工程实现均具备完整链路，可直观展示成员推理攻击在真实服务约束下的可行性和局限性。

### 1.6 作品边界

LabelScan 面向授权审计，不提供绕过鉴权、批量爬取第三方模型或攻击未授权服务的能力。系统输出的是风险判定而非确定事实，结果会受到目标模型精度、影子模型质量、样本分布、查询预算和校准样本选择的影响。在实际使用中，应将报告作为风险提示和人工复核依据，并结合模型训练记录、数据许可和业务场景进行综合判断。

## 第二章 作品设计与实现

### 2.1 总体设计思想

LabelScan 的总体设计遵循三个原则。

第一，攻击面约束真实化。系统默认目标模型只返回标签，不假设可以获得概率分数或梯度信息，使审计条件更接近真实商业 API。

第二，证据来源可解释化。系统不直接给出单一黑箱分数，而是把影子模型 loss、边界距离和波动系数分别呈现出来，并给出融合判定原因。

第三，系统交付工程化。成员隐私审计往往需要大量模型查询，单次任务可能持续数分钟甚至更久。因此系统必须具备任务排队、取消、状态轮询、报告持久化和服务自动恢复能力，而不能只依赖一次同步请求。

### 2.2 系统总体架构

LabelScan 采用前后端与模型服务解耦的架构，整体由用户入口、Go 审计后端、Python 模型服务、数据与报告层组成。

用户入口包括 Web 控制台和 Windows 启动器。Windows 启动器是一个轻量级 exe，不内置模型和审计逻辑，只负责打开服务器上的 HTTPS 控制台。这样普通用户无需安装 Docker、Python、CUDA 或模型权重，即可通过浏览器使用系统。

Go 审计后端是系统核心，负责接收审计请求、生成任务、管理队列、调用模型接口、执行边界攻击、计算风险信号并生成报告。后端同时提供 CLI 和 Web 服务能力，既可用于本地开发测试，也可作为服务器常驻服务运行。

Python 模型服务负责加载目标模型与影子模型。两个模型服务均通过 FastAPI 提供接口，目标模型至少提供 `/health`、`/predict`、`/predict_batch`，影子模型在 full 模式下额外提供 `/predict_logits`。通过接口解耦后，系统可将内置 CIFAR-10 示例模型替换为其他分类模型，只需保持输入格式和返回协议兼容。

数据与报告层负责读取样本、成员索引、阈值配置，并输出 JSON 与 HTML 报告。JSON 报告便于自动化分析，HTML 报告便于评审和人工复核。

### 2.3 模型接入协议

LabelScan 将目标模型抽象为 label-only 分类服务。目标模型接口只需要返回预测标签，边界攻击即可运行。为了提升边界搜索中的梯度估计效率，系统支持批量预测接口，将多个扰动样本合并为一次 HTTP 请求，减少网络往返开销。

表 2-1 模型服务接口

| 接口 | 服务 | 功能 |
| --- | --- | --- |
| `GET /health` | Target / Shadow | 返回服务状态、模型加载状态和运行设备 |
| `POST /predict` | Target / Shadow | 输入单张图片向量，返回预测标签和可选 logits |
| `POST /predict_batch` | Target / Shadow | 输入多张图片向量，批量返回预测标签 |
| `POST /predict_logits` | Shadow | 返回影子模型 logits，用于计算交叉熵 loss |

当前内置示例以 CIFAR-10 为数据集，输入为长度 3072 的展平图像向量，对应 \(3 \times 32 \times 32\) 的彩色图像。接入其他模型时，需要保证输入维度、归一化方式、类别空间和样本分布与目标任务一致。

### 2.4 双轨成员风险审计算法

LabelScan 将成员风险证据划分为两类：行为指纹和几何指纹。行为指纹来自影子模型 loss，几何指纹来自目标模型决策边界距离与稳定性。

#### 2.4.1 影子模型行为指纹

影子模型用于模拟目标模型在同类任务上的成员化行为。对于样本 \(x\) 和标签 \(y\)，影子模型输出 logits \(z\)，系统先计算 softmax 概率：

\[
p_i=\frac{e^{z_i}}{\sum_j e^{z_j}}
\]

然后计算交叉熵损失：

\[
L(x,y)=-\log p_y
\]

一般而言，若模型对某个样本更熟悉，正确类别上的概率会更高，交叉熵损失更低。因此，loss 越低，样本越可能呈现成员化行为。LabelScan 使用影子模型训练集和验证集统计得到阈值配置，包括严格阈值和宽松阈值。系统在运行时自动识别更严格的 loss 红线，避免阈值字段顺序或数值大小带来的误判。

影子模型信号被划分为三档：

表 2-2 影子模型信号分级

| 条件 | 信号等级 | 含义 |
| --- | --- | --- |
| loss 低于严格阈值 | RED | 样本表现出明显成员化行为 |
| loss 低于宽松阈值 | YELLOW | 样本存在一定成员化迹象 |
| loss 未触发阈值 | GREEN | 当前行为证据不足 |

#### 2.4.2 边界几何指纹

边界几何信号基于一个直观假设：训练成员样本往往处于模型更熟悉、更稳定的区域，距离决策边界可能更远；非成员样本或困难样本更可能靠近边界，在微小扰动下更容易改变预测标签。

对于样本 \(x\)，系统希望估计最小扰动距离：

\[
d(x)=\min_{\delta}\|\delta\|_2,\quad f(x+\delta)\ne f(x)
\]

由于目标模型只返回标签，系统采用 HopSkipJump 思想进行决策边界搜索。算法流程包括：

1. 初始化搜索：随机采样噪声点，寻找一个与原始标签不同的对抗点。
2. 二分逼近：在原图与对抗点之间二分搜索，将对抗点压到决策边界附近。
3. 梯度方向估计：在边界点附近采样多个随机方向，仅根据标签是否改变估计边界法向。
4. 几何步进：沿估计方向移动，并再次二分回到边界附近。
5. 查询预算控制：当达到最大查询次数或最大迭代次数时停止。

为提高效率，梯度估计阶段的多次扰动查询通过 `/predict_batch` 合并为批量请求。这样边界攻击仍然满足 Label-Only 条件，但可以显著减少 HTTP 往返次数。

#### 2.4.3 局部扰动稳定性

单次边界距离可能受到随机初始化和样本局部形状影响。LabelScan 对每个待测样本生成多个轻微扰动变体，对原图和变体分别估计边界距离，得到距离集合：

\[
D=\{d_0,d_1,\ldots,d_k\}
\]

系统计算平均边界距离：

\[
\bar d=\frac{1}{k+1}\sum_i d_i
\]

以及波动系数：

\[
CV=\frac{\sigma(D)}{\bar d}
\]

平均距离越大，说明样本可能离决策边界越远；波动系数越小，说明样本局部决策表现越稳定。二者共同构成边界几何指纹。

#### 2.4.4 动态阈值标定

固定阈值难以适配不同模型和不同数据分布。LabelScan 在审计前先抽取一批确定未参与目标模型训练的非成员样本作为“路人样本”，对其执行边界距离探测，并统计参考分布。系统根据参考距离集合生成距离阈值 \(\tau_D\) 和稳定性阈值 \(\tau_{CV}\)。当待测样本的平均边界距离超过 \(\tau_D\)，或波动系数低于 \(\tau_{CV}\) 时，会触发不同等级的几何风险信号。

动态标定使系统能够根据当前目标模型、当前数据分布和当前查询预算自适应调整判断标准，减少直接套用固定阈值带来的偏差。

#### 2.4.5 双轨融合判定

影子模型行为信号和边界几何信号分别从不同角度描述成员化痕迹。LabelScan 采用保守融合策略：当两个信号同时支持成员风险时，输出高风险；当仅单侧证据明显时，输出较高或中等风险；当目标模型对原图预测错误、边界距离异常但 loss 不支持、或证据不足时，系统会降级处理。

表 2-3 双轨融合规则

| 影子模型信号 | 边界几何信号 | 输出风险 |
| --- | --- | --- |
| RED | RED | 高成员风险 |
| YELLOW | RED | 高成员风险 |
| RED | YELLOW | 高成员风险 |
| GREEN | RED | 中等成员风险 |
| RED 或 YELLOW | GREEN | 较高成员风险 |
| GREEN | YELLOW | 较高成员风险 |
| GREEN | GREEN | 低成员风险 |

该策略的核心思想是避免因为单一信号偶然异常就直接给出强结论，同时保留对疑似样本的提示能力。

### 2.5 审计引擎实现

Go 审计引擎负责完成从样本到风险报告的主流程。对于每个样本，系统首先检查目标模型对原图的预测。如果目标模型无法正确识别原图，说明该样本在当前模型下不具备稳定审计基础，系统直接输出低风险或证据不足结论。

随后，系统调用影子模型接口获取 logits 并计算 shadow loss；再调用边界攻击模块估计原图及多个变体的边界距离；最后根据阈值和融合规则生成 `AuditResult`。结果中包含样本编号、参考成员标记、预测风险、风险等级、shadow loss、平均边界距离、波动系数和结论文本。

在 full 模式下，系统同时使用影子模型信号和边界信号；在 boundary-only 模式下，系统只依赖目标模型标签接口，适合作为外部模型初筛。boundary-only 模式证据弱于 full 模式，因此报告中会明确标注其为弱证据模式。

### 2.6 并发调度与任务队列

边界审计包含大量独立模型查询，尤其是在梯度估计阶段，一个样本可能需要成百上千次标签查询。LabelScan 使用 goroutine 和 channel 构建并发审计池，将样本级任务分发给多个 worker 执行。worker 数量可配置，默认标准模式使用 20 个 worker，快速 smoke 模式使用 2 个 worker。

Web 审计任务采用队列机制。系统同一时刻只运行一个审计任务，避免多个长任务同时抢占 GPU、端口和模型服务资源；新任务进入 FIFO 队列等待。页面提交任务后，后端立即返回任务编号，前端通过轮询接口查询任务状态，避免浏览器或代理在长时间同步请求中超时。

任务状态包括 queued、running、canceling、succeeded、failed 和 canceled。用户可以取消排队任务，也可以取消运行中的任务。运行中取消通过 context 传递取消信号，HTTP 客户端、边界攻击、审计引擎和 worker pool 会在当前模型请求返回后尽快停止。

### 2.7 Web 控制台与报告系统

Web 控制台提供完整审计交互界面，包括：

1. 选择审计模式：full 或 boundary-only。
2. 选择运行预设：smoke、standard、extended 或 custom。
3. 配置 Target API 与 Shadow API 地址。
4. 设置成员样本数、非成员样本数、并发数、最大查询数和迭代次数。
5. 检查目标模型与影子模型服务状态。
6. 提交审计任务、查看排队位置、取消任务并展示结果。

### 2.8 部署方案与安全控制

LabelScan 支持两类部署方式。

第一类是容器化部署。通过 Docker Compose 启动目标模型服务、影子模型服务和 Go Web 控制台，适合本地复现和标准化演示。生产环境中只建议对外暴露 Web 控制台或反向代理入口，Target 和 Shadow 模型服务应保持在 Docker 内网或服务器本机地址，避免公网直接访问模型推理接口。

第二类是无 sudo 服务器部署。对于没有管理员权限的共享 GPU 服务器账号，可以使用 Conda 环境直接运行 Python 模型服务和 Go 后端，并通过用户级脚本实现启动、停止、重启和 watchdog 自动拉起。模型服务默认绑定本机地址，公网访问通过 Cloudflare Tunnel 或 Nginx HTTPS 反向代理进入 Go 控制台。

服务器部署中，系统可将目标模型和影子模型放到指定 GPU 上运行，也可以拆分到两张 GPU 上降低单卡压力。由于模型服务启动后会加载权重并占用显存，即使没有用户访问也会保留模型显存，因此系统提供手动停止脚本和后台状态检查，便于在共享服务器上减少对其他用户的影响。

公网安全方面，系统建议叠加以下措施：

1. 仅对外开放 HTTPS Web 入口，不暴露目标模型和影子模型端口。
2. 使用 Basic Auth、访问 token 或登录系统限制访问者。
3. 对审计接口设置限流，防止高频任务刷取模型。
4. 使用任务队列限制同一时刻运行的审计任务数量。
5. 在报告中明确审计结果为风险提示，避免未经复核的结论被滥用。

### 2.9 Windows 轻量启动器

为了降低普通用户使用门槛，LabelScan 提供 Windows 启动器。启动器是单文件 exe，功能是打开服务器上的 LabelScan HTTPS 控制台。审计逻辑、模型权重、Python 环境和 GPU 计算均在服务器端完成，用户电脑只需要能访问域名即可。

这种设计具有三个优点。第一，用户无需配置复杂环境；第二，模型权重和数据不离开服务器，降低敏感资产泄露风险；第三，服务端可统一管理版本、队列、访问控制和算力资源，更适合多人短期使用或比赛展示。

### 3.4 性能测试

LabelScan 的性能瓶颈主要来自模型推理和 HTTP 查询，而不是本地数学计算。为提升整体效率，系统在两个层面进行优化：一是在 HSJA 梯度估计阶段使用批量预测接口，减少网络往返；二是在样本级审计阶段使用并发 worker 池，提高多样本审计吞吐。

本地基准测试中，向量距离、softmax、交叉熵、均值方差、噪声生成、插值和裁剪等基础运算均以微秒级完成。以 3072 维图像向量为例，L2 距离计算约 1.2 微秒，softmax 计算约 0.15 微秒，交叉熵计算约 0.18 微秒，向量裁剪约 2.1 微秒。1000 个样本的并发调度基准可在毫秒级完成，说明 Go 后端本身不会成为长任务瓶颈。

### 3.5 可靠性测试

LabelScan 针对长任务和共享服务器环境设计了可靠性机制。

第一，Web 审计采用异步任务机制。提交审计任务后，后端立即返回任务编号，前端轮询状态，避免 Cloudflare、Nginx 或浏览器因为长时间无响应而返回 HTML 错误页，从而导致前端 JSON 解析失败。

第二，系统支持任务取消。用户取消任务后，后端通过 context 向正在运行的审计流程传递取消信号，边界攻击和 HTTP 客户端在合适的检查点退出。

第三，后台服务提供 watchdog。若 Go 后端、模型服务或公网隧道异常退出，watchdog 可按周期检查并自动拉起，提升服务器无人值守运行能力。

第四，系统将模型服务限制在本机地址或内网地址。即使 Web 控制台对外开放，目标模型和影子模型端口也不会直接暴露给公网，降低模型接口被绕过控制台直接刷取的风险。

### 3.6 安全性分析

LabelScan 本身用于安全审计，但仍需要考虑工具被误用的风险。系统采取了以下设计。

一是权限边界明确。系统面向授权测试和自有模型审计，不包含绕过访问控制或非法获取训练数据的功能。

二是接口暴露最小化。对外只暴露 Web 入口，模型服务由后端调用，避免攻击者直接访问 `/predict_batch` 大量消耗算力。

三是结果解释保守。报告将输出描述为成员风险，而不是训练集归属事实，避免过度推断。

四是任务并发受控。队列机制保证同一时刻只有一个审计任务运行，减少共享 GPU 服务器上的资源冲突。

五是便于审计复核。每个结论都附带 shadow loss、边界距离和波动系数，审计人员可以根据证据进行人工判断。

### 3.7 局限性分析

LabelScan 仍存在一定局限。

第一，结果依赖目标模型质量。如果目标模型本身准确率较低，或对原图经常预测错误，边界距离和成员风险信号都会变弱。

第二，影子模型质量会影响 full 模式。影子模型若与目标模型分布差异过大，shadow loss 阈值可能无法稳定迁移。

第三，边界攻击查询成本较高。更充分的边界估计通常意味着更多模型查询和更长运行时间，需要在速度与稳定性之间取舍。

第四，当前示例主要面向 CIFAR-10 图像分类任务。虽然接口可扩展到其他分类模型，但不同数据模态需要重新设计输入预处理和阈值校准流程。

第五，公网部署需要额外访问控制。若面向多用户开放，应进一步加入登录系统、用户级配额、任务审计日志和更严格的限流策略。

## 第四章 创新性说明

### 4.1 面向 Label-Only 场景的双轨风险审计

LabelScan 针对只返回分类标签的严格黑盒条件设计审计流程，避免依赖目标模型概率分数和梯度信息。系统将影子模型行为指纹与目标模型边界几何指纹结合起来，使成员隐私审计能够在更贴近真实 API 的条件下运行。

### 4.2 动态阈值标定与样本级解释

系统不使用固定距离阈值直接判断成员风险，而是在审计前使用确定非成员样本进行现场标定，得到适配当前目标模型的距离阈值和稳定性阈值。每个样本的报告不仅给出风险等级，还保留 loss、距离、波动系数和证据摘要，使结果具备可解释性和可复核性。

### 4.3 高并发边界搜索工程实现

Label-Only 边界攻击的主要成本来自大量标签查询。LabelScan 在 HSJA 梯度估计阶段使用批量预测接口，并在样本审计阶段使用 goroutine worker pool，提升高频 HTTP 查询场景下的吞吐能力。该设计使系统能够从单次算法演示扩展为批量审计工具。

### 4.4 长任务队列与取消机制

成员隐私审计任务通常耗时较长，普通同步 Web 请求容易超时。LabelScan 将审计任务设计为可排队、可轮询、可取消的后台任务，既保证用户交互稳定，也避免共享服务器上多个任务同时抢占 GPU。该机制增强了作品作为实际软件系统的完整性。

### 4.5 服务器后端与轻量客户端结合

系统支持“服务器运行模型和后端，用户使用 exe 或浏览器访问”的交付方式。用户端不需要安装复杂依赖，模型和数据保留在服务器侧，既降低使用门槛，又减少模型资产泄露风险。这种部署形态适合短期比赛展示、课程实验和小范围授权审计。

### 4.6 安全部署意识

LabelScan 在设计中明确区分 Web 控制台和模型推理服务。目标模型与影子模型服务只在本机或内网暴露，公网入口可通过 HTTPS、访问控制和限流保护。系统同时保留后台状态检查和手动停止机制，适配共享 GPU 服务器环境。

## 第五章 总结

LabelScan 围绕 Label-Only 黑盒模型成员隐私审计问题，完成了从理论方法到工程系统的完整设计。系统在只能观察分类标签的条件下，结合影子模型行为信号和决策边界几何信号，对样本训练成员风险进行评估；通过动态阈值标定和双轨融合逻辑，提高了结果的可解释性和稳定性；通过 Go 并发后端、Python 模型服务、Web 控制台、任务队列、取消机制、报告系统和 Windows 启动器，形成了可部署、可演示、可复核的软件作品。

从测试结果看，LabelScan 能够在内置 CIFAR-10 示例任务上完成成员风险识别和报告生成，并在目标模型准确率有限的条件下保持较高查全能力。系统当前更适合作为隐私风险发现和安全审计辅助工具，而不是单独作为训练集归属判定依据。

后续工作可以从五个方向继续完善。第一，扩展到更多数据集和模型结构，包括更大规模图像模型、文本分类模型和多模态模型。第二，加入更正式的用户认证、任务配额、审计日志和访问令牌机制，提升公网部署安全性。第三，优化多 GPU 调度，让目标模型、影子模型和审计任务能够根据显存与负载自动分配。第四，引入更丰富的统计校准方法，提高不同数据分布下的风险判定稳定性。第五，完善报告导出能力，支持 Word、PDF 和更适合比赛答辩的可视化图表。

总体而言，LabelScan 将成员推理攻击、Label-Only 边界分析、并发工程和服务器化交付结合在一起，为人工智能模型隐私风险审计提供了一个完整、可运行、可解释的实践方案。
